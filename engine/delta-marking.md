# Delta 标记体系

> 从 SKILL.md 03 阶段提取。用于标记技术方案中每个文件的变更性质，驱动并行决策和审查策略。
> **完全通用**，不包含项目特定内容。

---

## 一、标记定义

| 标记 | 含义 | 对任务分批的影响 |
|:----:|------|------|
| **ADDED** | 新建文件（从无到有） | 可大胆并行，不依赖现有文件内容 |
| **MODIFIED-🔵追加** | 在文件末尾或指定位置追加内容，不修改现有代码 | 需先 `read_file` 了解末尾位置，但不会与其他 Agent 冲突 |
| **MODIFIED-🟡修改** | 修改文件中间的现有代码（精确 `replace_in_file`） | 串行风险高，需先探索再精确替换 |
| **MODIFIED-🔴替换** | 替换整个文件的核心逻辑（`write_to_file` 覆盖） | 串行化，必须独占该文件 |
| **REMOVED** | 删除文件 | 极少用，需同步更新编译单元注册（如 csproj） |

---

## 二、各标记对并行策略的影响

### ADDED（并行安全）
- 内容完全由方案决定，Agent 不需要读取任何已有文件
- 同一批次可无限并行（仅受资源限制）

### MODIFIED-追加（可并行读，但写需协调）
- **读取阶段**：多个 Agent 可同时 `read_file` 了解现有内容
- **写入阶段**：只要追加位置不同（如不同方法），可以并行；追加同一文件末尾时必须串行
- 判断原则：追加的内容是否可能互相依赖或发生行号冲突

### MODIFIED-修改（串行优先）
- 必须先完整读取文件，确定 `old_str` 的精确位置
- 两个 Agent 不可同时修改同一文件

### MODIFIED-替换（严格串行）
- 整文件替换，任何并行写入都会互相覆盖
- 必须独占执行，且执行前需用户确认

### REMOVED（谨慎执行）
- 删除前必须确认没有其他文件引用该模块
- 同步更新：编译单元注册文件（csproj/pom.xml/package.json 等）、import 引用文件

## 三、技术方案中的标准表格格式

所有技术方案文档的"代码改动范围"章节必须使用以下表格格式：

```markdown
| # | 文件路径 | Delta标记 | 说明 |
|---|---------|:---------:|------|
| 1 | Inf/Entity/CISMoniterDev.cs | **ADDED** | 新建Entity，映射 CIS_MONITER_DEV 表 |
| 2 | BLL/MonitorDeviceItemMappingMgr.cs | **MODIFIED-🔵追加** | 新增 6 个方法，不影响现有 5 个方法 |
| 3 | src/net/et/config_monitor/config_monitor.ts | **ADDED** | 新建 Net 层文件 |
| 4 | src/domain/triage_emergency_config.ts | **MODIFIED-🔵追加** | 追加 3 个方法到文件末尾 |
| 5 | BLL/OldFeatureMgr.cs | **REMOVED** | 功能合并到新 Mgr，删除旧文件 |
```

## 四、标记与代码审查规则的关联

| Delta 标记 | 代码审查激活的规则类型 |
|:----------:|---------------------|
| ADDED | 全量规则（新建文件：继承关系/命名/注解/结构完整性） |
| MODIFIED-追加 | 一致性规则（追加内容与已有代码风格/签名/模式一致） |
| MODIFIED-修改 | 局部规则（仅改动范围的正确性） |
| MODIFIED-替换 | 等同于 ADDED（整个文件视为新建） |
| REMOVED | 引用清理检查（确认没有悬空引用） |

---

## 五、命名冲突预检的触发条件

凡是 `MODIFIED-追加` 或向已有类追加方法/属性时，**必须**在 03 阶段和 04 阶段各执行一次命名冲突预检：

```
预检方式：search_content("目标方法名", "目标文件路径")
预检时机：
  - 03 阶段：技术方案输出前
  - 04 阶段 Step 1.5：任务清单审查时
预检结果：
  - 未找到同名 → 安全，继续
  - 找到同名 → 必须改名，使用更具体的名称
```
