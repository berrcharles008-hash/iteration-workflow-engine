# 自动复盘与进化机制

> 从 SKILL.md "自动复盘与进化机制" 章节提取。
> **核心理念**：Agent 不能等人指出错误才反思。必须在每个阶段结束时自动复盘，主动从问题中提炼规则，写入 Skill，持续进化。

---

## 一、复盘触发时机

| 时机 | 触发条件 | 复盘内容 |
|------|---------|---------|
| **修改门禁前** | Agent 准备调用写入工具但未加载 Skill | 检查为何门禁未触发 |
| **03 阶段开始** | 影响等级评估为 Normal/Emergency | 确认回滚方案已写入技术方案 |
| **03 阶段结束** | 命名冲突预检发现同名 | 记录为何方案设计时没预料到冲突 |
| **04 阶段结束** | Step 4.6 编译失败 / Lint 报错 | 记录所有编译错误的原因和修复方式 |
| **05 阶段结束** | `{frontend_dev_command}` 报错 / API 不可达 | 记录所有运行时错误 |
| **06 阶段结束** | 上线后发现问题 / 回滚方案被触发 | 全流程回溯 + 回滚方案有效性检验 |
| **用户指出流程错误** | 任何时候用户指出流程被绕过/跳过/门禁失效 | **立即复盘**，提炼模式，更新流程 |

---

## 二、复盘流程（五步）

```
Step 1: 识别根本原因
  ├── 是命名冲突？语法错误？逻辑错误？依赖遗漏？流程缺失？
  └── 一句话概括问题本质

Step 2: 判断是「个案」还是「模式」
  ├── 个案 → 仅修复代码，不更新流程
  └── 模式 → 代码修复 + 流程规则更新（下一步）

Step 3: 提炼预防规则
  ├── 在哪个阶段应该拦截？
  ├── 用什么检查手段？
  └── 把规则写入 Skill 对应阶段

Step 4: 更新 Skill 文件
  ├── SKILL.md：添加检查步骤 / 关键原则
  └── engine/ 或 project/：更新对应引擎/配置文件

Step 5: 写入迭代复盘日志
  └── {{ITERATIONS_DIR}}/{迭代编号}/lessons-learned.md
```

---

## 三、个案 vs 模式判断标准

| 特征 | 判断 |
|------|------|
| 只在这个迭代这个文件出现，其他迭代不会遇到 | 个案 |
| 任何类似操作（如"向已有类追加方法"）都可能遇到 | 模式 |
| 由特定技术约束导致（如某些语言不支持方法重载） | 模式 |
| 由流程缺失导致（如"没有命名冲突预检步骤"） | 模式（需补充流程） |

---

## 四、Skill 更新位置指南

| 问题类型 | 应更新的文件 |
|---------|------------|
| 流程缺失某个检查步骤 | `SKILL.md` 对应阶段 + `engine/workflow-engine.md`（未来） |
| 命名/架构规则 | `engine/naming-conflict-check.md` |
| 代码审查规则遗漏 | `project/code-review-rules.md` |
| 部署/构建问题 | `project/deploy-config.yaml` 或 `project/build-verify.yaml` |
| 复杂度评估偏差 | `engine/complexity-scoring.md` |

## 五、复盘日志模板

每次发现问题的复盘记录写入 `{{ITERATIONS_DIR}}/{迭代编号}/lessons-learned.md`：

```markdown
# 迭代复盘日志

## {YYYY-MM-DD} 复盘

### 问题：{简短标题}

| 维度 | 内容 |
|------|------|
| 发现阶段 | {哪个阶段、哪个步骤} |
| 根因 | {一句话描述根本原因} |
| 类型 | 个案 / 模式（{模式名称}） |
| 本应在哪个阶段拦截 | {应该在哪个阶段/步骤提前发现} |
| 预防措施 | {具体的预防规则，可操作} |
| Skill 更新 | {更新了哪些文件，更新了什么内容} |
```
