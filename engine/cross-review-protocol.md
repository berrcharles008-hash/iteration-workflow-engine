# 独立 Agent 交叉审查协议

> 从 `workflow-engine.md` 提取。定义独立 Agent 交叉审查的通用规范。
> 适用于 02（可选）/03（强制🟡🔴）/04（强制🟡🔴）阶段。

---

## 一、什么是交叉审查

独立 Agent（code-explorer 子代理）以"第三方 reviewer"视角审查主 Agent 的产出，消除作者盲区。

| 维度 | 主 Agent 自审 | 独立 Agent 交叉审查 |
|------|:---:|:---:|
| 执行者 | 产出文档的同一 Agent | 独立的 code-explorer 子代理 |
| 上下文 | 完整的对话上下文 + 代码库 | 仅读取文档和代码库文件（无对话上下文，无先入为主） |
| 盲区 | 存在——作者容易跳过"我认为没问题"的部分 | 无——不参与产出，无先入为主 |
| 适用场景 | 业务逻辑、设计意图验证 | 纯文件/规范/命名/一致性检查 |

---

## 二、触发时机

| 阶段 | 步骤ID | 触发条件 | 审查对象 |
|------|--------|:--:|------|
| 02-需求评审 | step-1x-cross-review | 🟡🔴（可选） | 需求文档格式完整性 |
| 03-技术方案 | step-3x-cross-review | 🟡🔴（强制） | 技术方案 L1/L2 |
| 04-开发实现 | step-1-5x-cross-review | 🟡🔴（强制） | 开发任务清单 |

> 🟢 简单需求跳过交叉审查——改动量极小（1-2文件、修bug、改label），独立Agent审查的 token 成本 > 收益。

---

## 三、审查维度（分级标准）

> 本分级用于独立 Agent 审查报告。03 阶段主 Agent 的方案评审使用另一套简化分级（🔴阻塞/📝注意），用途不同，见 `workflow-engine.md` §评审边界与通过标准。

| 级别 | 含义 | 处理方式 |
|:--:|------|---------|
| 🔴 致命 | 会导致编译失败/数据错误/安全漏洞/方案不可执行 | 必须修复 |
| 🟡 严重 | 逻辑矛盾/命名不一致/遗漏关键步骤 | 必须修复 |
| 🔵 轻微 | 格式问题/描述不清/可优化但不影响功能 | 可选修复 |
| 💡 建议 | 改进建议/更好的做法 | 记录，不强制 |

## 四、各阶段审查指令模板

### 4.1 03-技术方案 L1/L2 交叉审查

主 Agent 派发独立 Agent 时使用以下指令模板：

```
你是一个独立的代码审查Agent（"第三方 reviewer"），请以批判性视角审查以下技术方案文档，
找出主Agent（方案作者）可能遗漏的问题。

审查对象：{{方案文档路径}}
相关代码库：{{PROJECT_ROOT}}

## 审查清单

> 本 L1/L2 审查是独立 Agent 的**二次验证**，不替代方案输出前主 Agent 的命名冲突预检。

### L1：命名冲突扫描
- [ ] 方案中每个 ADDED 符号（类名/方法名/属性名），搜索目标文件中是否已存在同名符号
- [ ] 如果有同名冲突，标注冲突位置和符号名

### L2：关键符号验证（内部自洽 + 代码库交叉验证，最多5项）
- [ ] 方案"代码改动范围"表中每个引用类型（基类/接口/异常类/DTO），确认在代码库中存在
- [ ] 文件路径是否与实际项目结构一致
- [ ] 从改动范围表提取优先级1符号（异常类、IMgr方法）→ search_content 确认存在
- [ ] 从改动范围表提取优先级2符号（基类、路由名）→ search_content 确认存在

## 输出格式

请按以下格式输出审查报告，表格后附审查总结：

| # | 级别 | 位置 | 问题描述 | 建议修复 |
|---|:--:|------|---------|---------|
| 1 | 🔴/🟡/🔵/💡 | 方案文档 §X / 代码文件:行号 | ... | ... |

审查总结：共发现 X 个问题（🔴致命 Y / 🟡严重 Z / 🔵轻微 W / 💡建议 V）
```

### 4.2 04-任务清单交叉审查

主 Agent 派发独立 Agent 时使用以下指令模板：

```
你是一个独立的代码审查Agent（"第三方 reviewer"），请审查以下开发任务清单，
逐项对照技术方案检查是否有遗漏或不一致。

审查对象：{{任务清单路径}}
对照基准：{{技术方案路径}}
相关代码库：{{PROJECT_ROOT}}

## 审查清单

- [ ] 方案"代码改动范围"表中每个文件，任务清单中都有对应任务？
- [ ] 任务描述中的文件路径/类名/方法名是否与方案一致？
- [ ] ADDED 类型的任务描述中，是否明确声明了命名空间和继承关系？
- [ ] 任务依赖关系是否与方案的阶段划分一致？

## 输出格式

请按以下格式输出审查报告，表格后附审查总结：

| # | 级别 | 问题描述 | 方案引用 | 任务清单引用 |
|---|:--:|---------|---------|------------|
| 1 | 🔴/🟡/🔵/💡 | ... | 方案 §X | 任务 T... |

审查总结：共发现 X 个问题（🔴致命 Y / 🟡严重 Z / 🔵轻微 W / 💡建议 V）
```

### 4.3 02-需求格式完整性检查（🔴 强制 / 🟡 可选）

主 Agent 派发独立 Agent 或组装外部模型 prompt 时使用以下指令模板：

```
你是一个独立的需求审查Agent，请检查以下需求文档的格式完整性（不评审业务合理性）。

## 审查对象

以下为需求文档完整内容（如有多份文档，用 --- 分隔）：

{{嵌入需求文档的完整文本内容}}

## 审查清单

- [ ] 需求标题、背景、范围是否完整？
- [ ] 是否有明确的验收标准？
- [ ] 边界场景是否描述清楚？
- [ ] 是否有遗漏的必填字段？
- [ ] 各文档之间是否自洽（无内部矛盾）？
- [ ] 架构分层（如 engine/ 与 project/）职责是否清晰无重叠？

## 输出格式

| # | 级别 | 问题描述 |
|---|:--:|---------|
| 1 | 🔴/🟡/🔵/💡 | ... |

审查总结：共发现 X 个问题（🔴致命 Y / 🟡严重 Z / 🔵轻微 W / 💡建议 V）
```

> ⚠️ 注意：主 Agent 替换 `{{嵌入需求文档的完整文本内容}}` 时必须嵌入文档的实际内容（通过 Read 读取），不能仅提供文件路径。外部模型无文件系统访问权限，只给路径会导致审查结果为空或推理链失控。

## 五、主 Agent 合并规则

独立 Agent 返回审查报告后，主 Agent 按以下规则合并处理：

| 独立Agent发现 | 处理 |
|:--|------|
| 🔴 致命 | 主 Agent **立即修复**，修复后主 Agent 自检确认（不重新启动独立 Agent） |
| 🟡 严重 | 主 Agent **立即修复**，修复后主 Agent 自检确认（不重新启动独立 Agent） |
| 🔵 轻微 | 主 Agent 评估后**选择性修复**，记录处理决定 |
| 💡 建议 | 记录，不阻塞阶段推进 |
| 独立Agent标记但实际无误 | 主 Agent 标注"误报"，记录理由 |

> ⚠️ **边界场景**：修复后**不需要重新启动独立 Agent 审查**——主 Agent 自检确认即可（否则会无限循环）。
>
> ⚠️ **误报裁定权**：用户拥有最终裁定权。如用户认为某项误报标记不当，主 Agent 必须重新评估。

### 合并输出格式

```
## 交叉审查结果（独立 Agent）

| # | 级别 | 发现问题 | 处理 |
|---|:--:|---------|------|
| 1 | 🔴 | Xxx不存在 | ✅ 已修复 |
| 2 | 🟡 | 路径不一致 | ✅ 已修复 |
| 3 | 🔵 | 描述可优化 | 📝 已记录，不阻塞 |
| 4 | 💡 | 建议使用Xxx模式 | 📝 已记录，暂不处理 |

独立Agent交叉审查完成：共 4 项，🔴0 🟡0 🔵1 💡1（除误报外已全部处理）
```

## 六、并行执行约定

- 主 Agent 先启动独立 Agent（异步 task），再立即执行自身的自审步骤——两者**同时启动**，互不阻塞
- 两份审查结果都返回后，主 Agent 执行合并
- 合并后的修复完成后，才进入下一步：
  - **03 阶段**：进入 step-4-review-gate（评审边界检查）
  - **04 阶段**：进入 step-1-6-user-confirm（用户确认任务清单）

> ⚠️ **步骤清单中的序号不代表执行顺序**：step-3x-cross-review / step-1-5x-cross-review 在步骤表中排在 step-3-self-review / step-1-5-review 之后，但这仅是为了编号有序。实际执行时，独立 Agent 互审与主 Agent 自审**必须同时启动**（主 Agent 先 `task` 派发独立 Agent，再立即执行自身自审）。03/04 阶段均适用此规则。

## 七、复杂度门禁

| 阶段 | 🟢 简单 | 🟡 中等 | 🔴 复杂 |
|------|:--:|:--:|:--:|
| 02-需求评审 | ❌ 跳过 | 可选 | ✅ **强制**（外部模型优先） |
| 03-技术方案 | ❌ 跳过 | ✅ 强制 | ✅ 强制 |
| 04-开发实现 | ❌ 跳过 | ✅ 强制 | ✅ 强制 |

> 02 阶段 🔴 复杂级强制使用外部审查模型（优先调用 `review-gateway.py`，失败降级独立 Agent 审查）。🟡 中等级可选。🟢 简单级跳过。
> 03/04 涉及代码级审查（命名冲突/路径一致性/接口签名验证），必须强制互审兜底。

## 八、外部审查模型路由（v2.0）

> **设计目标**：审查使用第三方模型（如 SenseNova DeepSeek-V4），编码使用默认工具模型。第三方不可用时自动熔断降级，回退到默认 Agent 审查。

### 8.1 架构

```
主 Agent（默认模型，负责编排）
  │
  ├─ Step R1: 组装审查 prompt → 写入临时文件
  ├─ Step R2: Bash("python scripts/review-gateway.py --prompt-file <temp>")
  │            │
  │            ▼
  │       review-gateway.py（Python 脚本）
  │            │
  │            ├─ 读取 project/review-models.json（模型优先级列表）
  │            ├─ 读取 runtime/review-circuit.json（熔断状态）
  │            ├─ 循环: 按优先级试每个可用模型
  │            │   ├─ 熔断未过期 → 跳过
  │            │   ├─ 无 API Key → 跳过
  │            │   ├─ POST /v1/chat/completions
  │            │   │   ├─ 成功 → 清除熔断 → 返回 {"ok":true, "text":"..."}
  │            │   │   └─ 失败 → 打开熔断(TTL=5min) → 试下一个
  │            │   └─ 全部失败 → 返回 {"ok":false}
  │            └─ stdout: JSON 结果
  ├─ Step R3: 解析 stdout JSON
  │   ├─ ok==true  → 使用 text 作为审查结果，标注 "外部模型: {used}"
  │   └─ ok==false → ★ FALLBACK: 使用默认 Agent 自行审查，标注 "外部模型不可用，已降级"
  └─ Step R4: 合并审查结果 → 按 §五 合并规则处理
```

### 8.2 配置（project/review-models.json）

```json
{
  "version": 1,
  "default_ttl_minutes": 5,
  "models": [
    {
      "name": "sensenova-v4-flash",
      "base_url": "https://token.sensenova.cn/v1",
      "api_key_env": "SENSENOVA_API_KEY",
      "model": "deepseek-v4-flash",
      "priority": 1
    },
    {
      "name": "sensenova-6.7-flash-lite",
      "base_url": "https://token.sensenova.cn/v1",
      "api_key_env": "SENSENOVA_API_KEY",
      "model": "sensenova-6.7-flash-lite",
      "priority": 2
    }
  ]
}
```

- **priority**：数字越小越优先。熔断后自动跳到下一个。
- **api_key_env**：API Key 走环境变量（如 `SENSENOVA_API_KEY=sk-xxx`），不在此文件硬编码。
- **新增模型**：在 `models` 数组末尾追加对象即可，无需改脚本。

### 8.3 熔断机制（runtime/review-circuit.json）

```json
{
  "version": 1,
  "circuits": {
    "sensenova-v4-flash": {
      "open_since": "2026-07-18T14:30:00.123456+00:00",
      "fail_count": 3,
      "last_error": "HTTPError: 503 Service Unavailable",
      "ttl_minutes": 5
    }
  }
}
```

- 调用失败 → 自动写入熔断记录，TTL=5分钟
- TTL 未到期 → `review-gateway.py` 跳过该模型
- TTL 到期 + 下次调用成功 → 自动清除熔断
- 此文件**仅由 Python 脚本读写**，Agent 不应直接操作

### 8.4 Agent 执行步骤

当触发交叉审查（02可选/03强制/04强制）时，主 Agent 自动检测外部模型配置并按本路由执行：

```
## 审查步骤（外部模型路由版）

Step 1: 检查 review-models.json 是否存在且 models 非空？
  → 不存在或为空 → 跳过，使用默认子 Agent 审查（零影响）
  → 存在且有模型 → 继续

Step 2: 组装审查 prompt（参考 §四 模板），写入临时文件
  → 写入路径: {skill_dir}/runtime/review-prompt-{stage}.txt

Step 3: 调用网关
  → 使用 Bash 工具执行:
     python scripts/review-gateway.py
       --prompt-file "{skill_dir}/runtime/review-prompt-{stage}.txt"
       --skill-dir "{skill_dir}"

Step 4: 解析 stdout JSON
  → 若 ok==true:
       - 将 text 作为审查报告
       - 输出标注: "📡 外部审查模型: {used} ({model})"
    若 ok==false:
       - ⚠️ FALLBACK: 启动 code-explorer 子 Agent 执行审查（原 §四 流程）
       - 输出标注: "⚠️ 外部模型全部不可用，已降级为默认 Agent 审查"

Step 5: 合并结果（按 §五 规则处理审查发现）
```

### 8.5 触发条件与降级

| 条件 | 行为 |
|------|------|
| 未配置 `project/review-models.json` 或 models 为空 | 跳过，使用默认子 Agent 审查 |
| 环境变量 `{api_key_env}` 未设置 | 跳过，使用默认子 Agent 审查 |
| Python 不可用 / 脚本不存在 | 跳过，使用默认子 Agent 审查 |
| review-gateway 返回 ok==true | ✅ 使用外部模型结果 |
| review-gateway 返回 ok==false | FALLBACK → 默认子 Agent 审查 |

> **渐进式采用**：不强制。仅当配置了 `review-models.json` 且 API Key 可用时自动启用。不配置 → 行为与现有完全一致（零影响）。

### 8.6 前置依赖

| 依赖 | 要求 | 检查方式 |
|------|------|---------|
| Python 3 | 本机可用 | `python --version` 返回 3.x |
| 环境变量 | `{api_key_env}` 已设置 | `os.environ.get(...)` |
| 网络 | 可连通 `{base_url}` | 脚本内 HTTP 调用失败即熔断 |

### 8.7 与独立 Agent 审查的并行关系

外部模型路由**替换** code-explorer 子代理审查（而非并行）。两种模式二选一：

| 模式 | 审查者 | 适用场景 |
|------|--------|---------|
| 默认模式（无外部模型） | code-explorer 独立 Agent（Claude） | 未配置 review-models.json |
| 外部模型模式 | review-gateway（第三方模型） | 已配置且有可用 API Key |

主 Agent 的自审步骤不受影响，始终执行。
