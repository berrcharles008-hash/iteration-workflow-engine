# 门禁协议

> 定义迭代工作流的三道门禁规则：修改门禁、迭代门禁、步骤门禁。
> 本文件是门禁规则的**唯一真相源（Single Source of Truth）**。
> 路径占位符（`{{ITERATIONS_DIR}}` 等）运行时从 `project/project.manifest.yaml` 注入。

---

## 一、修改门禁协议

### 核心理念

任何代码修改都有风险。即使改动再小（改 label、改文案），也需要影响评估、回测用例、阶段审批。Agent 不得在迭代流程外直接修改代码文件。

**触发条件**：Agent 准备调用写入类工具（replace_in_file / write_to_file / delete_file）时。
**优先级**：★ 门禁检查优先于一切用户指令。即使用户明确说"帮我修改XX"，也必须先过门禁。

### 门禁规则

```
Agent 准备修改文件
    ↓
读取 runtime/ACTIVE → 单文件内容（O(1)，仿 Git HEAD 指针）
    ├── ACTIVE 不存在 或 内容为 "none"
    │   └── 回退扫描 runtime/*.state.yaml（兜底，防止 ACTIVE 失同步）
    │       ├── 无 iteration_status=in_progress 的迭代 → 确认 BLOCK，输出：
    │       │   ╔══════════════════════════════════════════════════╗
    │       │   ║  🛑 修改门禁：当前无活跃迭代                    ║
    │       │   ║                                                  ║
    │       │   ║  所有代码修改必须经过迭代工作流。                ║
    │       │   ║  请选择：                                        ║
    │       │   ║  1️⃣ 为此需求创建新迭代（推荐）                   ║
    │       │   ║  2️⃣ 将需求并入当前某个已有迭代                   ║
    │       │   ╚══════════════════════════════════════════════════╝
    │       │   等待用户选择，Agent 不得自行绕过。
    │       │
    │       └── 有 in_progress 的迭代（ACTIVE 失同步）
    │           → 修复 ACTIVE 写入正确迭代ID → 从顶部重新进入
    │
    └── ACTIVE 内容为迭代ID → 读取 runtime/{ID}.state.yaml
        ├── state.yaml 不存在 → ACTIVE 损坏，回退扫描（同上）
        ├── iteration_status 为 completed 或 paused（ACTIVE 失同步）
        │   └── 修复 ACTIVE="none" → 回退扫描
        └── iteration_status 为 in_progress → 正常门禁流程
            ├── current_phase != "04-开发实现" → BLOCK，输出：
            │   ╔══════════════════════════════════════════════════╗
            │   ║  🛑 修改门禁：当前不处于开发阶段                ║
            │   ║                                                  ║
            │   ║  当前阶段：{current_phase} - {phase_status}      ║
            │   ║  代码修改仅在 04-开发实现 阶段允许。             ║
            │   ║  请选择：                                        ║
            │   ║  1️⃣ 继续推进当前迭代到 04 阶段                   ║
            │   ║  2️⃣ 将本修改作为独立需求新建迭代                 ║
            │   ╚══════════════════════════════════════════════════╝
            │
            └── current_phase == "04-开发实现" → 校验文件迭代归属（★ 2026-06-29 故障后新增）
                ├── 文件路径含 docs/iterations/{id}/ → id 与活跃迭代ID是否一致？
                │   ├── 一致 → ALLOW，继续正常执行
                │   └── 不一致 → BLOCK，输出：
                │       ╔══════════════════════════════════════════════════╗
                │       ║  🛑 修改门禁：文件不属于当前活跃迭代            ║
                │       ║                                                  ║
                │       ║  要修改的文件属于迭代 {file_id}                  ║
                │       ║  该迭代状态：{file_iter_status}                 ║
                │       ║  当前活跃迭代：{active_id}（04阶段）             ║
                │       ║                                                  ║
                │       ║  已完成迭代的交付物不能直接修改。                ║
                │       ║  请选择：                                        ║
                │       ║  1️⃣ 重新打开 {file_id} 迭代（回退到04阶段）      ║
                │       ║  2️⃣ 将本修改作为独立需求新建迭代                 ║
                │       ╚══════════════════════════════════════════════════╝
                │
                └── 文件路径不在 docs/iterations/ 下（普通源代码）→ ALLOW
```

## 二、迭代门禁协议

> 解决问题：当前迭代尚未完成（如卡在 04 阶段），用户直接要求"进入新迭代"，可能导致旧迭代残留、状态丢失。
> 核心机制：Agent 在响应「开新迭代」类请求前，必须检查是否有活跃的 `runtime/*.state.yaml`。

### 门禁规则

```
用户请求"新迭代"/"下一个迭代"/"开始新需求"
    ↓
读取 runtime/ACTIVE → 单文件内容（O(1)，仿 Git HEAD 指针）
    ├── ACTIVE 不存在 或 内容为 "none"
    │   └── 兜底扫描 runtime/*.state.yaml（确保 ACTIVE 未失同步）
    │       ├── 不存在 in_progress/paused 的迭代 → ALL_CLEAR：继续创建新迭代
    │       └── 存在（ACTIVE 失同步）→ 修复 ACTIVE 写入迭代ID → BLOCK
    │
    └── ACTIVE 内容为迭代ID → 读取 runtime/{ID}.state.yaml
        ├── state.yaml 不存在 → ACTIVE 损坏，回退扫描
        └── state.yaml 存在 → 检查 iteration_status
            ├── completed → 修复 ACTIVE="none" → ALL_CLEAR
            └── in_progress 或 paused → BLOCK，输出：
                ╔══════════════════════════════════════════════╗
                ║  🛑 迭代门禁：存在未完成的迭代                ║
                ║                                              ║
                ║  迭代编号：{ITERATION_ID}                     ║
                ║  整体状态：{iteration_status}                 ║
                ║  当前阶段：{current_phase} - {phase_status}   ║
                ║  已完成阶段：{从 phase_history 提取}          ║
                ║  待执行阶段：{phase_history 中缺失的 01~07}   ║
                ║                                              ║
                ║  ❌ 无法跳至新迭代。                         ║
                ║                                              ║
                ║  可选操作：                                   ║
                ║  1️⃣ 继续当前迭代（进入下一待执行阶段）        ║
                ║  2️⃣ 归档当前迭代（需走完 05→06→07 全部阶段）    ║
                ║  3️⃣ 强制跳过（需说明理由，当前迭代标记        ║
                ║     paused，下次访问时重新激活）              ║
                ╚══════════════════════════════════════════════╝

                请选择 1️⃣ / 2️⃣ / 3️⃣：
```

### 强制跳过协议

用户选择选项 3️⃣ 时：
- Agent 将当前 state.yaml 的 `phase_status` 设为 `paused`，并写入 `pause_reason`
- 下次用户提及该迭代编号时，Agent 自动检测 `paused` 状态并提示恢复
- 禁止静默归档——archive 操作必须经过 06 阶段

### 迭代删除门禁协议

> 解决问题：Agent 收到"删除迭代"指令后不经校验直接执行，导致已归档迭代的完整知识资产被不可逆删除。

```
用户请求"删除迭代" / "删除 XXX 迭代"
    ↓
强制步骤 1：定位迭代状态文件
    ├── 无状态文件 → 提示"未找到该迭代"
    └── 有状态文件 → 进入步骤 2
    ↓
强制步骤 2：检查迭代状态
    ├── *.state.archived.yaml（已归档）
    │   └── 🛑 BLOCK：禁止删除！
    │       "❌ 该迭代已归档，包含需求文档、技术方案、代码实现、
    │        上线记录等完整知识资产，不可删除。"
    │       → 对话结束，不接受理由
    │
    ├── iteration_status = "in_progress"（进行中）
    │   └── ⚠️ 警告："该迭代正在进行中，删除将丢失所有进度。
    │       包括 N/N 任务中的已完成部分。确定要删除吗？"
    │       → 用户确认后再执行
    │
    └── phase_status = "paused"（暂停）
        └── ⚠️ 警告："该迭代处于暂停状态。确定要删除吗？"
            → 用户确认后再执行
```

**核心规则**：**已归档 = 不可删除，不接受任何理由。**（在 [workflow-engine.md#迭代门禁协议](workflow-engine.md#迭代门禁协议) 章节也有引用）

## 三、步骤门禁协议

> 解决问题：用户的口令直接穿透流程，导致强制步骤被跳过。
> 核心机制：Agent 将每个阶段的步骤写入 `state.yaml` 的 `phase_steps`，在**进入阶段时一次性初始化**；后续响应**仅在满足重武装触发条件时拦截**，避免每轮都扫描的 token 浪费。

### 三-A. 阶段转换评审门禁（Phase Transition Review Gate）

> 解决问题：评审不通过时，Agent 仍可推进到下一阶段。
> 核心机制：从 02 或 03 阶段推进到下一阶段前，检查 `phase_history[N].review_gate.result`。

```
推进阶段 M → M+1（M 为 02 或 03）
    ↓
检查 phase_history[M].review_gate.result
    ├── 不存在 review_gate（legacy） → 允许推进（向后兼容）
    ├── "passed" → 允许推进
    ├── "conditionally_passed"（仅 02） → 允许推进
    └── "rejected" → BLOCK：
            "🛑 评审未通过，不能进入下一阶段"
            "阶段：{M}，原因：{rejection_reason}"
            "请选择：1️⃣ 重新执行评审  2️⃣ 强制跳过"
```

- 强制跳过写法：在 `review_gate` 上追加 `overridden_by` / `override_reason` / `overridden_at`
- 完整规则见 [state-protocol.md](state-protocol.md) §3.1 review_gate

### 三-B. 阶段推进确认门禁（Phase Advance Confirmation Gate）

> 解决问题：Agent 产出阶段文档后，在未经用户明确确认的情况下直接修改 `current_phase` 推进阶段。
> 事故案例：014 迭代，Agent 产出 03-技术方案后在同一轮次直接将 `current_phase` 从 "03" 推进到 "04"，跳过用户评审确认。
> 核心机制：修改 `current_phase` 字段前，必须验证本轮用户的指令中包含明确的"阶段通过/确认"意图。

#### 触发条件

Agent 准备修改 `runtime/{id}.state.yaml` 的 `current_phase` 字段（即将 M → M+1）。

#### 门禁规则

```
Agent 准备修改 current_phase（M → M+1）
    ↓
检查：当前轮次用户的指令中是否包含"阶段确认通过"意图？
    ├── 明确通过语：
    │     "通过" / "确认" / "OK" / "没问题" / "可以" /
    │     "进入XX阶段" / "进入0X" / "下一阶段" / "继续到下一阶段" /
    │     "方案没问题"/"评审通过"
    │   → ALLOW，允许推进 current_phase
    │
    └── 不包含以上明确通过语（如用户只说"继续"、或Agent在产出文档
        后同一轮次直接推进）→ 🛑 BLOCK：
            ╔══════════════════════════════════════════════════╗
            ║  🛑 阶段推进门禁：缺少用户确认                    ║
            ║                                                  ║
            ║  正在尝试推进：{M} → {M+1}                        ║
            ║  当前阶段 {M} 产出已生成，但未经用户评审确认。   ║
            ║                                                  ║
            ║  ★ 原则 #8：逐阶段确认，不可跳过。               ║
            ║  Agent 不得在产出文档后直接推进 current_phase。   ║
            ║  请等待用户明确确认（"通过"/"没问题"/"进入下一阶段"等）║
            ║  后再推进。                                      ║
            ╚══════════════════════════════════════════════════╝
            终止推进，保持 current_phase = M。
```

#### 与 三-A 的关系

| 门禁 | 检查内容 | 触发时机 |
|------|---------|---------|
| **三-A** 评审门禁 | 评审是否通过（`review_gate.result`） | 推进 02→03 或 03→04 时 |
| **三-B** 确认门禁 | 用户是否明确确认（本轮指令含通过语） | **任何** current_phase 变更时 |

两条门禁都通过，才允许推进阶段。三-A 检查"质量"，三-B 检查"流程合规"。

#### "继续"歧义处理

| 用户指令 | Agent 应执行 | 推进 current_phase？ |
|---------|-------------|:--:|
| "继续"（无上下文） | 在当前阶段内继续推进工作 | ❌ |
| "继续到下一阶段" / "进入04" | 推进到下一阶段 | ✅（先过三-A和三-B） |
| "通过" / "没问题" | 标记当前阶段完成 → 询问是否推进 | ❌（需等推进指令） |
| "通过，进入下一阶段" | 标记完成 + 推进 | ✅ |

#### 豁免

以下情况不触发此门禁：
- `current_phase` 在同一阶段内不变更（如仅更新 `phase_status`、`last_updated`、`phase_steps[N].status`）
- 06→07 推进（06 已含部署确认，等同用户确认）

#### 输出格式参考

**本门禁执行前**，Agent 应先按 `workflow-engine.md` 的「阶段完成提示模板」输出清晰的可操作选项（A.进入下一阶段 / B.审查本轮产出 / C.其他指令），引导用户选择。

> 两个阶段的区分：阶段完成提示模板是**输出规范**（Agent 对用户说的最后一句话），本门禁是**检查规范**（收到用户回复后检查是否含明确通过语）。模板在前，门禁在后，顺序不可颠倒。

### 三-C. 阶段推进前的步骤结清门禁（Step Settlement Gate）

> 解决问题：阶段内强制步骤（mandatory step）仍为 pending，Agent 即因用户"已发布/已验证"等事实陈述推进到下一阶段，导致强制步骤被静默越过（014 迭代 P-028 事故）。
> 核心机制：修改 current_phase 前，检查当前阶段 phase_steps 中所有 `mandatory: true` 的步骤是否均为 completed / skipped。

#### 触发条件
Agent 准备修改 current_phase（M → M+1），且 M 阶段存在 `mandatory: true` 的步骤。

#### 门禁规则
扫描 phase_steps 中 mandatory=true 的步骤：
    ├── 全部为 completed 或 skipped → ALLOW
    └── 存在 pending 的 mandatory 步骤 → 先结清再推进：
         ① 该步骤确已实质完成（如用户已手动发布 = 真实环境验证）
            → 置 status: completed + completed_reason: "{理由}"
         ② 该步骤被有意跳过（如用户确认无需执行）
            → 置 status: skipped + skip_reason: "{理由}"
         ③ 结清全部 pending mandatory 步骤后，才允许修改 current_phase
        ⚠️ 禁止带着 pending 强制步骤直接推进（等同静默跳过，违反用户跳过覆盖规则）。

#### 与 三-B 的关系
| 门禁 | 检查内容 | 触发时机 |
|------|---------|---------|
| **三-B** 确认门禁 | 用户是否明确确认（含通过语） | 任何 current_phase 变更时 |
| **三-C** 结清门禁 | 本阶段强制步骤是否均已 completed/skipped | 任何 current_phase 变更时（含 06→07） |
两条门禁都通过，才允许推进阶段。三-B 检查"流程确认"，三-C 检查"步骤结清"。

> **06→07 推进说明**：三-B 确认门禁豁免 06→07（因 06 已含部署确认），但 **三-C 结清门禁正常执行**——必须检查 06 阶段所有 mandatory 步骤（含 step-4-archive-check）均已 completed/skipped，才能推进到 07。两条门禁独立执行，三-C 不受三-B 豁免影响。

### 门禁规则（条件重武装）

```
进入阶段 → 将本阶段所有步骤写入 state.yaml phase_steps（一次性初始化）
    ↓
每次用户请求 → 检查"重武装触发条件"（满足任一即拦截）：
    ├── 触发条件：
    │     ① 用户指令跳步：未确认 03/04 即要求写码
    │     ② Agent 自检跳步：当前 mandatory step 仍为方案审查/任务确认却要落码
    │     ③ 恢复后首轮：state-protocol 恢复逻辑已显式输出"当前应执行步骤"
    ├── 命中 → 扫描 phase_steps，找到第一个 pending 强制步骤 → BLOCK：
    │         "⚠️ 步骤门禁：{step-name} 尚未完成（强制步骤），
    │          无法跳至您请求的操作。是否先执行 {step-name}？"
    ├── 用户要求跳过 → 写入 status: skipped + skip_reason
    └── 未命中 → ALL_CLEAR：正常响应用户请求
```

> **恢复逻辑硬规则**：state-protocol 恢复后必须显式输出"当前应执行步骤"，防止步骤信息内化后丢失（见 [state-protocol.md](state-protocol.md) 第八节）。

### 用户跳过覆盖

强制步骤可被跳过，但必须显式确认并记录理由：
- Agent 不得静默跳过
- 跳过理由写入 `state.yaml` 的 `skip_reason` 字段

## 四、门禁豁免

以下操作**不触发**修改门禁：
- 读取文件（read_file / search_content / search_file）
- 创建迭代文档目录和文档文件（docs/iterations/ 下的 .md 文件）
- 修改 `.codebuddy/skills/` 下的 Skill 自身进化文件

## 五、影响等级分类（ITIL 对齐）

> 门禁通过后，Agent 须在进入 03-技术方案 前对修改评估影响等级，决定执行深度。

| 等级 | 代号 | 判据 | 允许跳过阶段 | 回滚方案 |
|:----:|------|------|:-----------:|:------:|
| 🔵 标准 | Standard | 改文案/修样式/加注释，影响 <2 文件 | 02评审 | 可选 |
| 🟡 正常 | Normal | 改逻辑/增字段/增页面，影响 2-5 文件 | 无 | **必须** |
| 🔴 紧急 | Emergency | 修线上 Bug/数据修复，影响 1-3 文件 | 02评审、05部分 | **必须** |

**等级评估流程**：
1. Agent 评估后输出等级 + 理由
2. 用户确认或手动调整等级
3. 等级写入 `runtime/*.state.yaml` 的 `impact_level` 字段

## 六、回滚方案模板（🟡🔴 等级强制）

> Normal 和 Emergency 等级修改必须在 03-技术方案 中包含以下回滚计划。
> Standard 等级可选。

```markdown
### 回滚计划

| 维度 | 内容 |
|------|------|
| 回滚触发条件 | {什么情况下需要回滚，如：上线后报错/Bug 增加/性能劣化} |
| 回滚操作步骤 | 1. {git revert <commit>} 2. {恢复配置/回滚SQL} 3. {验证} |
| 回滚验证方法 | {如何确认回滚成功，如：执行冒烟测试/检查报表数据} |
| 预估回滚耗时 | {分钟数} |
| 回滚责任人 | {执行者} |
```
