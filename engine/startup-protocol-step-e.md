# 启动协议 — Step E 每日工作日志

> 引用 `engine/startup-protocol.md` 的 Step A-D 通用启动流程。
> 本条仅为项目级补充：对话结束前写入每日工作日志（Claude Code 特有，CodeBuddy 已有自己的日志机制）。

---

## Step E：对话结束前写入每日工作日志（★ 强制）

> **触发时机**：每次对话结束前（用户发出结束信号、Agent 完成当前任务准备退出、或对话自然终止时）。
> **独立于 state.yaml**：不论是否触发了 state.yaml 的更新，只要本次对话有工作产出，就必须写入。

### E.1 日志路径

```
.claude/memory/YYYY-MM-DD.md
```

日期使用当前对话日期。文件不存在则创建，存在则追加。

### E.2 写入内容

每次追加一个 `## {迭代标题} — {简要说明}` 段落，内容至少包含：

| 字段 | 说明 |
|------|------|
| **修改了哪些文件** | 列出本次对话中所有写入/修改的文件路径 |
| **遇到了什么问题** | Bug、编译错误、框架陷阱等 |
| **遗留了什么** | 未完成的任务、待用户执行的操作、待确认的事项 |

> 内容控制在 200~500 字。不需要长篇大论，但要让 Agent 和用户都能在 30 秒内掌握今日全貌。

### E.3 写入后验证（★ 强制）

写入完成后必须执行以下两项验证：

```
验证 1：读取刚写入的文件，确认段落已追加成功（不是空写就丢）
验证 2：检查内容完整性 — 修改文件列表、问题、遗留事项三项是否齐全
         任一缺失 → 补全后再退出
```

### E.4 与 state.yaml 的关系

| 协议 | 用途 | 触发条件 |
|------|------|---------|
| `state.yaml` | 结构化进度跟踪 | 阶段/任务完成时 |
| `memory/YYYY-MM-DD.md` | 按天聚合的工作摘要 | **每次对话结束前** |

两者是**独立轨道**：
- state.yaml 更新但对话未结束 → 不写 memory（等对话结束统一写）
- 对话结束但 state.yaml 未触发更新 → **仍需写 memory**（今天有活，就该记）

> 示例：修改了 3 个文件但任务清单还没打勾（补丁修改），state.yaml 没变，但 memory 必须记录今天的修改。

### E.5 示例

```markdown
## 015 系统分级配置 — 04开发实现完成

**修改文件**：
- `back-end/.../Inf/Dtos/DictTriageGradeDto.cs` — 追加 9 字段
- `front-end/.../config_triage_grade.ts` — 与 upgrade 同步
- ...（共 19 项）

**遇到问题**：
- `HasBusinessFields()` 错误返回 true → 改为 false
- 第 8 次编译才通过，之前 7 次逐步修复了 Entity Field 构造等问题

**遗留**：
- 用户需手动执行 3 个 SQL 脚本
- 部署到测试服务器验证
```
