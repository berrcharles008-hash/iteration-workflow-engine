## ★ 阶段完成提示模板（强制，所有阶段通用）

> 解决问题：Agent 完成阶段工作后，只说"等待确认推进"，用户不知道具体要确认什么、
> 有哪些可选操作。本模板要求 Agent 在每个阶段全部工作完成后，输出固定格式的提示语，
> 给用户清晰的可操作选项，降低认知负担。

### 模板定义

```
## ✅ {阶段名} 完成

**本轮已完成**：
- {事项1}
- {事项2}
- {事项3}

**接下来你可以选择**：

A. **进入 {下一阶段名}** — {一句简述下一阶段做什么}
B. **审查本轮产出** — {一句简述审查什么，如"打开文档逐段过目"}
C. **其他指令** — 比如补充修改、新增需求等

请选择 A/B/C 或直接告诉我你的想法：
```

### 各阶段选项说明

| 阶段（当前 → 下一阶段） | A 选项描述 | B 选项描述（审查内容） |
|:---:|-----------|----------------------|
| 01 → 02 | 进入需求评审，审阅需求文档 | 打开 01-需求记录.md 逐段过目 |
| 02 → 03 | 进入技术方案，设计实现方案 | 打开 02-需求评审.md 确认评审记录 |
| 03 → 04 | 进入开发实现，生成任务清单并编码 | 打开 {{DOC_03_TECHNICAL}} 审查方案细节 |
| 04 → 05 | 进入测试验证，生成用例并执行测试 | 打开变更文件和任务清单审查代码 |
| 05 → 06 | 进入发布上线，部署到测试服务器 | 打开 05-测试验证报告.md 审查测试结果 |
| 06 → 07 | 进入迭代回顾，总结并沉淀经验 | 打开 06-发布上线记录.md 审查上线记录 |

> **与门禁协议的关系**：本模板在阶段推进确认门禁（gate-protocol.md §三-B）**之前**执行。
> Agent 先输出此模板引导用户选择，用户选择 A 后，再走门禁协议检查用户是否明确确认。
> 两个步骤不冲突：模板是**输出规范**，门禁是**检查规范**。

### 示例（04→05 过渡）

```
## ✅ 04-开发实现 完成

**本轮已完成**：
- 3 项后端代码修改完成，编译通过
- 新增 1 个前端页面，构建无报错
- 代码审查通过，无阻塞级问题

**接下来你可以选择**：

A. **进入 05-测试验证** — 自动生成测试用例并执行验证
B. **审查本轮代码** — 打开变更文件逐段过目
C. **其他指令** — 比如补充修改、新增需求等

请选择 A/B/C 或直接告诉我你的想法：
```

### 执行规则

1. **放置在阶段最后的产出文档完成之后**，输出模板前须确认本阶段所有步骤已完成
2. **A/B 选项为固定格式**（"进入 XX 阶段" + "审查本轮产出"），C 选项为开放兜底
3. **B 选项（审查本轮产出）的审查方式**应根据阶段动态适配（见上表）
4. 用户选择 A 后，进入门禁检查流程（gate-protocol.md §三-B）
5. **禁止行为**：Agent 不得在输出模板后不等用户选择就自行推进 `current_phase`

> 解决问题：后期阶段发现问题需要回退改代码时，Agent 思维惯性"问题→代码"跳过了文档同步，
> 导致上游阶段产出文档与最终代码不一致（文档腐化）。
> 核心机制：回退到 M 阶段时，必须先检查 M 到 N-1 所有阶段的文档是否需要同步更新。

### 门禁规则

```
N 阶段发现问题 → 需要回退到 M 阶段执行代码修改
    ↓
强制步骤 1：扫描 M 到 N-1 各阶段的产出文档
    ├── 03-技术方案 → 设计方案是否有变？（如从 string.Format → 参数化）
    │   ├── 有变 → BLOCK：必须先更新 03-技术方案，再进入 M 阶段
    │   └── 无变 → 通过
    ├── 02-评审记录 → 设计变更是否引出新评审点？
    │   ├── 有 → 提示用户是否需要补充评审
    │   └── 无 → 通过
    └── 01-需求文档 → 需求理解是否有变？
        └── 同上
    ↓
强制步骤 2：更新完所有受影响的文档后，方可进入 M 阶段执行代码修改
    ↓
M 阶段完成后，重新按阶段顺序推进至 N 阶段（如 04→05）
```

### 典型场景

| 发现阶段 | 回退到 | 必检文档 | 示例 |
|:---:|:---:|------|------|
| 05 发现Bug | 04 | 03 | 配置SQL参数化改造 → 03 技术方案需新增 SqlParameterBinder 设计 |
| 04 编码发现 | 03 | 无（已在03） | 方案漏了某个边界情况 → 直接在03补充 |
| 05 验证失败 | 03 | 03→04 | 方案设计错误导致无法通过测试 → 重做03→04→05 |

### 输出要求

回退时 Agent 必须明确输出：
```
🔄 阶段回退：05 → 03（原因：SQL参数化方案变更）
   已检查受影响文档：
   ✅ 03-技术方案：需更新（ADDED SqlParameterBinder、MODIFIED BLL实现代码示例）
   ✅ 02-评审记录：无影响
   ✅ 01-需求文档：无影响
   待更新完文档后进入 03 阶段。
```

## 复杂度分级（★ 所有阶段启动前必须评估）

> 详细评估算法与自适应模板参见 [complexity-scoring.md](complexity-scoring.md)（11 因素评分 + 阈值映射 + 各等级执行模板 + 自适应对照表）。

| 等级 | 判据 | 典型场景 |
|:----:|------|---------|
| 🟢 **简单** | 修改 1-2 个已有文件，不涉及新表/新 API/新页面 | 改 label、增删下拉选项、修 bug、调样式 |
| 🟡 **中等** | 修改 3-5 个已有文件，或新增 1 个页面无后端改动 | 新增配置页面（复用已有 API）、修改现有页面逻辑 |
| 🔴 **复杂** | 新增数据库表、新增 API 接口、前端三层都变、6+ 文件 | 新功能模块、全栈改造 |

> 各等级在各阶段的执行方式差异，详见 [complexity-scoring.md](complexity-scoring.md) 三、各等级下各阶段的执行模板。

## 阶段二：02-需求评审

**目标**：评审需求文档，识别遗漏、矛盾、风险点。

**输入**：`01-需求分析与设计/01-需求记录.md`

**执行方式**：主 Agent 以 reviewer 视角审阅，提出问题，用户逐项确认或驳回

**🟡 评审必问 4 问模板**（防误升级 / 防漏评审）：

1. **范围**：本次需求覆盖哪些功能 / 数据 / 边界？
2. **边界**：不在范围内的内容是什么？与现有模块如何切分？
3. **风险**：技术风险、数据风险、兼容性风险有哪些？
4. **是否需要 03 方案**：是否超出 🟢 简单预判、需进入 03-技术方案？（第 4 问用于防止 🟡 被误升级到 03）

## 阶段三：03-技术方案

**目标**：产出完整技术方案文档，覆盖前后端设计、API 协议、代码改动范围。

**执行方式**：主 Agent 分析代码库，结合需求产出方案。需要时派生 code-explorer 探索代码。

**★ 复杂度升级硬规则**（03 探索段强制）：探索过程中若发现需求范围超出 01 阶段预判（如：本判 🟢 实则需新增多文件 / 跨层改动 / 引入新依赖），Agent **必须显式提示用户并建议升级复杂度**，不得静默硬撑；升级须用户确认后写入 `state.yaml` 的 `complexity` / `complexity_score`。

**产出**：`03-技术方案/{{DOC_03_TECHNICAL}}`，必须包含：
- 现状与痛点
- 改造方案（DDL、数据流、页面设计）
- 代码改动范围（Delta 标记体系，见 `engine/delta-marking.md`）— 含 ADDED/MODIFIED 标注
- API 接口协议（路由、参数、返回值、接口签名）— **必须**
- 开发阶段与任务拆解

**🔴 复杂方案的代码示例策略**（与项1 关键符号验证协同，降 token / 提可靠）：
- 方案聚焦「**改动范围表 + 接口签名**」；完整前后端代码示例改为**可选/按需**，仅在关键算法或新模式时附最小可运行片段，避免与实际编码偏离及 token 浪费
- 🟢/🟡：聚焦改动范围表 + 接口签名即可，代码示例可选
- **04 Team Agent 派发硬规则**：使用 Team Agent 派发时，task prompt **必须**含「改动范围表 + 接口签名 +（关键算法/新模式的最小可运行片段）」；普通 CRUD 仅需接口签名

**关键原则**：
- API 接口协议必须在方案中明确定义，前后端对齐
- 标注每个文件是"新增"还是"修改"以及改动类型（ADDED/MODIFIED-追加/修改/替换）
- **★ 命名冲突预检**（方案输出前强制）：详见 `engine/naming-conflict-check.md`

### ★ 自主审查（强制，输出方案后、交付前执行）

> 方案输出完成后，Agent **必须自行执行以下两层审查**，全部通过后才能向用户提交确认。
> 审查必须读取实际代码文件进行交叉比对，不得仅凭记忆判断。
> 审查必须输出明确的通过/发现问题的结论。

| 层级 | 检查项 | 执行方式 |
|:--:|------|------|
| L1 | **命名冲突**：新增的方法/类型/变量名是否与目标文件中已存在的符号重复 | 搜索目标文件中的同名符号 |
| L2 | **关键符号验证**：方案中引用的关键符号是否在实际代码库中存在 | 从改动范围表提取符号，最多验证 5 项（优先级排序见下） |

**L2 关键符号验证规则**（★ 合并原 L2 内部自洽 + L3 代码库交叉验证）：

1. **从方案文档的「代码改动范围」表提取所有新增/修改的符号**（类名、方法名、异常类、路由名等）
2. **按优先级排序，最多取 5 项**：
   - 优先级 1（必验）：异常类（`XxxException`）、接口签名（`IMgr` 方法）
   - 优先级 2（必验）：继承基类（`: SJMgrBase` / `: api`）、路由名（`[WebApi("...")]`）
   - 优先级 3（按需验）：前端类名大小写、参数绑定方式、权限前提
3. **每项用 `search_content` 在代码库中搜索一次**，确认实际存在
4. **超过 5 项时**：只验前 5 项，其余标注"⚠️ 建议后续验证"

> **★ 禁止行为**：Agent 完成方案输出后，**不得直接问用户"是否确认通过"**。必须先完成 L1→L2 两层审查，输出审查结论后，再邀请用户确认。
>
> 事故案例：2026-06-30-007 迭代中，Agent 仅做 L1 审查就邀请用户确认，遗漏 L2（DeleteResult 未列入）和 L3（BusinessException 不存在、前端类名大小写错误、DBMS_LOCK 权限未标注），导致方案存在编译级错误。

> **为什么合并 L2+L3**：原 L3 有 6 项验证清单，每项都要 `search_content`，token 消耗巨大且 Agent 容易疲劳跳过。合并后用"优先级 + 最多 5 项"控制验证深度，确保最重要的符号一定被验到。

### ★ 独立 Agent 交叉审查（强制，🟡🔴）

> 主 Agent 先启动独立 Agent（code-explorer 子代理），再立即执行自身自审（两者同时启动，互不阻塞）。
> 独立 Agent 不受作者盲区影响，能发现主 Agent 自审遗漏的问题。

**审查内容**：L1 命名冲突 + L2 关键符号验证（与自审相同维度，独立执行，最多5项）。

**执行流程**：
1. 主 Agent 启动独立 Agent，传入审查指令模板（详见 `engine/cross-review-protocol.md` §4.1）
   - 运行时将 `{{方案文档路径}}` 替换为 `{ITERATIONS_DIR}/{ID}/03-技术方案/{{DOC_03_TECHNICAL}}` 的绝对路径
   - `{{PROJECT_ROOT}}` 替换为项目根目录绝对路径
2. 独立 Agent 读取方案文档 + 搜索代码库 → 输出分级审查报告（🔴致命/🟡严重/🔵轻微/💡建议）
3. 主 Agent 按合并规则处理（🔴🟡→必须修复，🔵→可选，💡→记录）

**并行约定**：独立 Agent 与主 Agent 自审**同时启动**（主 Agent 先 `task` 派发独立 Agent，再立即执行自身自审），互不阻塞。⚠️ 步骤清单中的序号不代表执行顺序。

**适用复杂度**：🟡🔴（🟢 跳过，改动量极小的交叉审查 token 成本 > 收益）。

> 为什么自审 + 互审双保险？自审覆盖业务理解和设计意图（独立Agent做不到），
> 互审覆盖盲区和遗漏（自审做不到）。两者互补，不是替代。

### ★ 异步时序分析（强制，涉及跨组件/跨生命周期数据流时执行）

> 解决问题：Agent 在 03 方案阶段只做"表面代码逻辑走读"（看通路、看分支），
> 没有深入到框架机制层分析异步时序，导致真正的根因延迟到 04 开发阶段才被发现。
> 具体事故案例和当前项目框架的 API 映射见项目层时序指南：`project/{{frontend_framework}}-async-timing-patterns.md`

**触发条件**：方案涉及以下任一场景时，必须执行异步时序分析：
1. 父子组件/模块通过框架引用机制交互（子组件引用、跨组件数据操作）
2. 跨生命周期钩子的数据依赖（挂载/创建/副作用钩子/侦听器）
3. 子组件/模块初始化时有异步 API 请求或数据加载
4. 父组件在自身初始化阶段操作子组件的数据或方法

**分析步骤**：
1. **画异步时序图**（Mermaid sequenceDiagram）：标注每个 `await` 断点、数据传播点、跨组件调用点
2. **标注数据前提条件**：每个跨组件调用点需要哪些数据已就绪
3. **验证前提条件**：对照时序图，逐一检验数据就绪时间 vs 调用时间
4. **未满足的前提 → 增加等待步骤**（如等待子组件异步初始化完成、等待下一渲染周期）

**检查清单**（自主审查时逐项确认；具体 API 映射见项目层时序指南）：
- [ ] 子组件是否有异步初始化（框架挂载/创建钩子中的 await 调用）？→ 父组件不能假设其已完成
- [ ] 父组件修改数据后是否立即操作子组件？→ 需等待下一渲染周期
- [ ] 子组件是否有异步初始化方法？→ 父组件必须先等待它完成
- [ ] 模拟延迟下时序是否仍然正确？→ 关键路径建议用延迟验证

> 当前项目（Vue2）的具体 API 映射和事故案例见：`project/vue2-async-timing-patterns.md`

### ★ 评审边界与通过标准（强制，自主审查完成后执行）

> 解决问题：Agent 在 03 方案阶段反复评审，每次都发现新问题（编译错误、UX、权限、配置...），
> 把本该属于"代码审查"的问题提前到"方案评审"，导致评审无限循环、效率低下。
> 核心机制：明确方案评审的检查范围和通过条件，超出范围的问题不在方案阶段处理。

#### 评审检查项（仅以下4项，不扩展）

| # | 检查项 | 不通过的标准（致命问题） |
|---|--------|----------------------|
| 1 | 架构合理性 | 数据流断裂、分层错误、依赖方向错误、缺失关键依赖 |
| 2 | 设计决策正确性 | 策略选择有明显更优方案且未说明理由、关键决策依据错误 |
| 3 | 安全漏洞 | SQL注入、权限绕过、数据泄露、删除未做安全防护 |
| 4 | 遗漏关键场景 | 未处理异常路径、未考虑并发/超时/部分失败 |

**不在方案评审范围内**（留待04阶段编码/编译/代码审查处理）：
- 变量命名、类型签名、import 语句缺失
- 编译错误（如同名成员、类型不匹配）
- 注释措辞、文档一致性
- UX 细节（loading 状态、空值显示）
- 配置确认（NLog 目标、权限框架）
- 性能微调

#### 通过条件

```
通过条件：4 项检查全部无致命问题
不通过条件：任一检查发现致命问题
```

非致命问题归类为"📝 实现注意"，记录在方案末尾但不阻塞进入04阶段。

#### 问题分级（简化为2级，去掉模糊的"建议修复"）

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| 🔴 阻塞 | 架构缺陷/安全漏洞/设计错误/遗漏关键场景 | 必须修复后才能进入04 |
| 📝 注意 | 实现细节/UX优化/配置确认/性能微调 | 记录，04编码时处理，不阻塞 |

#### 评审轮次限制

| 阶段 | 评审类型 | 最大轮次 |
|------|---------|:--:|
| 03 方案 | 架构评审 | 1轮（用户可手动追加1轮） |
| 04 编码后 | 代码审查 | 1轮（编译器+人工） |
| 05 测试前 | 集成审查 | 1轮 |

**方案阶段默认只做1轮评审。** 如用户未明确要求再次评审，不主动发起。

#### 评审输出格式

```
## 03 方案评审结论

| 检查项 | 结果 |
|--------|------|
| 1. 架构合理性 | ✅/❌ {说明} |
| 2. 设计决策正确性 | ✅/❌ {说明} |
| 3. 安全漏洞 | ✅/❌ {说明} |
| 4. 遗漏关键场景 | ✅/❌ {说明} |

📝 实现注意（04阶段处理）：
- {非致命问题1}
- {非致命问题2}

评审结论：通过 / 不通过（{原因}）
```

### ★ 修复验证（Fix Verification，强制，每次修复后执行）

> 解决问题：Agent 修复评审问题后直接输出"全部完成"，未验证修改是否完整、是否引入新问题、
> 相关文档是否同步，导致修复本身引入新缺陷（如改名后同名冲突、描述与代码不一致）。
> 核心机制：每次修复（无论03方案还是04代码）后，必须执行3项验证，全部通过才算完成。

#### 适用范围

**所有阶段的修复操作**，包括但不限于：
- 03 方案评审问题修复后
- 04 编译错误修复后
- 04 代码审查问题修复后
- 05 测试失败修复后

#### 验证清单（3项，缺一不可）

| # | 验证项 | 执行方式 | 不通过的处理 |
|---|--------|---------|------------|
| 1 | **修改完整性** | 搜索所有引用点，确认都同步修改（如方法改名→搜索所有调用点） | 补全遗漏的引用点 |
| 2 | **未引入新问题** | 检查改动点是否与现有代码/方案冲突（同名成员、类型不匹配、逻辑矛盾） | 修复新引入的问题 |
| 3 | **文档同步** | 相关描述/注释/表格是否与改动后的代码一致 | 同步更新文档 |

#### 执行流程

```
发现问题 → 修复 → 修复验证（3项）
              ↓
              ① 修改完整性：搜索所有引用点 → 全部同步？
                 ├── 否 → 补全 → 重新验证①
                 └── 是 → ②
              ② 未引入新问题：检查同名/类型/逻辑冲突？
                 ├── 有 → 修复 → 重新验证②
                 └── 无 → ③
              ③ 文档同步：描述/注释/表格一致？
                 ├── 否 → 同步更新 → 重新验证③
                 └── 是 → ✅ 验证通过
              ↓
          输出验证结论 → 继续
```

#### 输出格式

```
## 修复验证（Fix Verification）

| # | 验证项 | 结果 | 说明 |
|---|--------|:--:|------|
| 1 | 修改完整性 | ✅ | 搜索 N 处引用，全部同步 |
| 2 | 未引入新问题 | ✅ | 无同名/类型/逻辑冲突 |
| 3 | 文档同步 | ✅ | 相关描述已同步更新 |

验证结论：通过
```

**禁止行为**：
- ❌ 修复后直接输出"全部完成"而不执行验证
- ❌ 只验证修改点本身，不检查引用点和文档
- ❌ 验证发现新问题后不修复就继续

> 事故案例：2026-06-30-007 迭代03阶段，Agent 修复 P2（统一命名 EstimateCount）后未验证引用点，
> 导致数据属性 `estimateCount` 与方法 `estimateCount()` 同名（P9），直到下一轮评审才被发现。

**交付标准**：三层自主审查通过 + 4项评审检查通过 + 修复验证通过 + 方案覆盖所有改动点 + 前后端对齐 + 用户确认"方案通过"

### Step 2：创建 Team 并派发任务

> 派发前读取 `project/agent-prompt-examples.md`，按模板组装 Agent prompt。
> 并行决策算法见 `engine/team-agent-strategy.md`。

每个 Agent 完成后，立即更新 `开发任务清单.md` 对应任务状态为 ✅ 已完成。
**同时按 `engine/state-protocol.md` 更新 `runtime/` 状态文件。**

### Step 4：完整性校验

- 检查所有文件是否存在于指定路径
- 检查 import/using 引用是否正确
- 运行 `read_lints` 检查语法错误

### Step 4.5：后端编译单元注册（★ 强制）

> 注册规则见 `project.manifest.yaml` → `compilation_units`。

执行后输出：**"编译单元注册完成：{按实际层数输出}"**

### Step 4.6：后端构建验证（★ 强制）

> 编译命令见 `project/build-verify.yaml` → `backend.build_command`。

| 结果 | 处理 |
|------|------|
| ✅ 通过 | 继续进入 Step 5 |
| ❌ 有 error | 停止，修复后重试（最多 3 次） |
| ⚠️ warning | 记录但不阻塞 |

输出：**"后端构建验证：{success_criteria} — 通过 ✅"**

### Step 4.7：Spec 合规验证（★ 强制，🔴复杂级 + 有 ADDED 文件时）

> 阻止"任务清单描述正确但代码写错"类问题。Step 4.6 编译通过 ≠ 代码符合 Spec。

**触发条件**：迭代为 🔴 复杂级，且任务清单中有 ADDED 类型文件。

| 检查项 | 验证方式 |
|--------|---------|
| **命名空间** | `read_file` 读取实际文件，检查 `namespace` 声明是否与 Spec 标注一致 |
| **继承关系** | 检查 `class Xxx : BaseClass` 是否与 Spec 一致（基类约定见 `project/context-conventions.md` → 二、后端类型约定） |
| **目录位置** | 检查文件所在物理目录是否与 Spec 路径一致 |
| **类名后缀** | 类名后缀规则见 `project/context-conventions.md` → 二、后端类型约定（2.2 类名后缀约定） |

**执行**：从 Spec 的「代码改动范围」表格提取 ADDED 文件，逐一打开实际文件比对。比对失败 → ❌ 阻塞，修正后重跑 Step 4.6 + Step 4.7。

输出：**"Spec 合规验证：N 项全部一致 ✅"** 或 **"发现 M 项不一致，已修正 → 重跑编译"**。

### Step 5：自动代码审查

> 详细规则见 `project/code-review-rules.md`

```
Step 5.1: 读取开发任务清单 → 提取变更文件 + 操作类型
Step 5.2: 按目录匹配规则矩阵触发条件，未改动层跳过
Step 5.3: 根据复杂度等级确定规则激活/跳过
Step 5.4: 逐条执行规则 → 记录通过/建议/问题

```

**交付标准**：所有任务 ✅ 完成，无 🔴 阻塞级审查问题，用户确认"开发完成"

## 阶段七：07-迭代回顾（★ 所有复杂度强制）

**目标**：每个迭代结束后的强制回顾，反思过程、沉淀经验、形成改进行动项。

**对应 Scrum**：Sprint Retrospective

**触发条件**：06 阶段归档完成（`iteration_status = "completed"`, `current_phase = "07"`）

**回顾对象**：**过程**（工作方式、流程、协作、工具），不是**产品**（那是 05 Review 干的）

**适用复杂度**：全部等级都不能跳过——Bug 修复迭代更必须做

### 与现有复盘机制的关系

| 现有机制 | 07-回顾 |
|---------|--------|
| `lessons-learned.md`（被动复盘） | 主动回顾（每个迭代强制） |
| 原 06 阶段 `step-3-lessons`（正向提炼） | 合并吸收：正向模式提炼迁移到 07 的 step-5 |
| 触发条件：出错时 | 触发条件：每个迭代结束 |
| 产出：单条记录 | 产出：回顾报告（结构化） |

### 各复杂度等级的执行模板

| 等级 | 执行深度 | 最小产出 | 时间盒 |
|:----:|---------|---------|:--:|
| 🟢 简单 | 精简版 | 回顾记录（追加到 `01-需求记录.md` 末尾，300字以内） | 5分钟反思 |
| 🟡 中等 | 标准版 | 独立 `07-迭代回顾报告.md`（含问题清单+改进措施） | 15分钟反思 |
| 🔴 复杂 | 完整版 | 独立 `07-迭代回顾报告.md` + 模式库更新 + lessons-learned沉淀 | 30分钟反思 |

### 回顾报告模板

#### 精简版（🟢 简单）

直接追加到 `01-需求记录.md` 末尾：

```markdown
## 回顾（{日期}）

### 本次迭代总结
- 做了什么：{一句话}
- 花了多久：{估算}
- 有没有返工：有/无（{说明}）

### 改进点
- {0~1个改进点}
```

#### 标准版/完整版（🟡 中等 + 🔴 复杂）

独立文件 `07-迭代回顾报告.md`，模板见 `engine/templates/phase-07-迭代回顾报告.md`。

### 步骤清单（进入阶段时写入 phase_steps）

| 步骤ID | 步骤名称 | 强制 | 🟢 | 🟡 | 🔴 |
|--------|---------|:--:|:--:|:--:|:--:|
| step-1-summary | 迭代总结（做了什么/做对了什么/可改进什么） | ✅ | 精简 | 标准 | 完整 |
| step-2-issues | 问题清单（本次迭代遇到的问题） | ✅ | 精简 | 标准 | 完整 |
| step-3-patterns | 模式沉淀（写入 lessons-learned 模式库） | — | 跳过 | 按需 | ✅ |
| step-4-actions | 行动项（改进措施落实到人） | — | 跳过 | ✅ | ✅ |
| step-5-lessons | 正向模式提炼（从原 06 step-3 迁移过来） | ✅ | ✅ | ✅ | ✅ |
| step-6-archive | 回顾归档（更新 state.yaml 完成 07，释放 ACTIVE） | ✅ | ✅ | ✅ | ✅ |

### Step 5：正向模式提炼（★ 强制，60秒反思）

**不依赖用户提醒，Agent 主动执行**：

```
问题：本次迭代有没有"以前不确定怎么做、这次首次跑通"的路径？

判断标准（满足任一即为正向模式）：
  ├── 之前没有先例，这次摸索出了正确做法
  ├── 之前靠猜测，这次验证了某个技术决策是对的
  └── 之前流程模糊，这次明确了某个步骤的正确顺序

有 → 写入 engine/lessons-learned.md 模式库，type 标记为 success（P-00N）
无 → 跳过，不强制生成
```

> 区别于错误模式（来自失败复盘），正向模式来自成功路径，同样有复用价值。

### Step 6：回顾归档

- 将 `iteration_status` 保持为 `"completed"`，`phase_status` 设为 `"completed"`
- **释放 ACTIVE 指针**：将 `runtime/ACTIVE` 写入 `"none"`
- 此时迭代才算**真正完成**

**产出**：`07-迭代回顾报告.md`（🟡/🔴）或 `01-需求记录.md` 末尾追加（🟢）

**交付标准**：回顾报告完成，ACTIVE 释放，用户确认"回顾完成"

### ★ 流程变更说明

```
原流程：01 → 02 → 03 → 04 → 05 → 06 → (结束，06 归档时释放 ACTIVE)
新流程：01 → 02 → 03 → 04 → 05 → 06 → 07 → (结束，07 归档时释放 ACTIVE)

06 阶段变更：
- step-3-lessons（正向模式提炼）从 06 移除，迁移到 07 step-5
- 06 归档后 iteration_status 设为 "completed"，current_phase 推进到 "07"
- 06 归档不再释放 ACTIVE 指针

07 阶段结束后：
- 07 归档后才释放 ACTIVE（写入 "none"）
- 防止新迭代在回顾完成前抢占
```
