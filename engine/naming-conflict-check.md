# 命名冲突预检规则

> 从 SKILL.md 03 阶段提取。防止在已有类中追加同名方法/属性，避免编译失败或运行时冲突。
> 经验来源：历史上多次发生向同一类追加同名方法的事故。

---

## 一、触发条件

凡技术方案中涉及以下操作时，**必须**执行命名冲突预检：

| 操作类型 | 是否必须预检 |
|---------|:---------:|
| 向已有类中追加新方法 | ✅ 必须 |
| 向已有类中追加新属性/字段 | ✅ 必须 |
| 新建文件（全新类） | ❌ 不需要（类名本身用下面的类名检查） |
| 新增 API 端点 | ✅ 必须（同类中不能有重复 `[WebApiMethod]` 名） |
| 新增路由/菜单项 | ✅ 必须（Id 和 path 不能重复） |

---

## 二、检查清单

| 检查项 | 搜索方法 | 命令示例 |
|--------|------|---------|
| 前端方法名 | 搜索目标文件中是否已有同名方法 | `search_content("{MethodName}", "{{TARGET_FILE}}")` |
| 前端类名 | 搜索目标目录是否已有同名类 | `search_content("class {ClassName}", "{{TARGET_DIR}}")` |
| 后端方法签名 | 搜索 IMgr 接口和 BLL 实现类中是否已有同名方法 | `search_content("{MethodName}\(", "{{INTERFACE_FILE}}")` |
| 后端 API 端点 | 搜索 WebApi 类中是否已有同名 `[WebApiMethod]` | `search_content("{MethodName}", "{{WEBAPI_FILE}}")` |
| 路由 path | 搜索是否已有相同的 path | `search_content("/{route_path}", "{{ROUTER_FILE}}")` |
| 菜单 Id | 搜索是否已有相同 Id | `search_content("{menu_id}", "{{MENU_FILE}}")` |

---

## 三、检查规则

1. **发现同名 → 必须改名**：使用更具体的名称（如 `GetDeviceManageList` 替代泛泛的 `GetList`）
2. **方法签名相似（参数不同但名称相同）**：某些语言不支持重载（如 TypeScript），必须改名；支持重载的语言（如 C#）允许重载但易混淆，优先使用不同名称
3. **即使方案文档中的方法名"看起来没问题"，也必须执行实际 `search_content` 验证**，不依赖主观判断

---

## 四、执行时机（两次，缺一不可）

| 时机 | 阶段 | 执行动作 |
|------|------|---------|
| **第一次** | 03-技术方案输出**之前** | 对所有 MODIFIED-追加 类型的目标文件，搜索所有新方法名 |
| **第二次** | 04-开发实现 Step 1.5 任务清单审查 | 再次确认，防止两次迭代并行时冲突 |

---

## 五、冲突发现后的处理流程

```
发现冲突：目标文件中已存在 {MethodName}
    ↓
Step 1: 在技术方案中修改该方法的命名
Step 2: 更新所有引用该方法的调用处（跨层一致性）
Step 3: 在 lessons-learned.md 记录冲突原因
Step 4: 重新执行本预检（确认新名称不冲突）
```
