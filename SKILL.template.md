**任何代码修改前必须先过门禁。** 门禁协议完整规则见 `engine/gate-protocol.md`。

## 七阶段概览

| 阶段 | 产出 | 关键动作 |
|------|------|---------|
| 01-需求分析 | 需求文档.md | 读 Spec 活文档 → 与用户确认需求 |
| 02-需求评审 | 评审记录.md | reviewer 视角审阅 → 提问题 → 用户决议 |
| 03-技术方案 | 技术方案设计.md | 代码探索 → 方案 → Delta标记 → 命名冲突预检 |
| 04-开发实现 | 开发任务清单.md + 代码 | 清单生成 → 自动审查 → 用户确认 → Team Agent → 构建 → 代码审查 |
| 05-测试验证 | 测试报告.md | 构建验证 → 用例生成 → 自动执行 → 降级手动 |
| 06-发布上线 | 上线记录.md | 编译 → 部署 → Spec 活文档更新 → 状态归档 |
| 07-迭代回顾 | 迭代回顾报告.md | 总结反思 → 问题清单 → 模式沉淀 → 归档释放 |

## 关键原则

0. **★ 修改门禁强制**：任何代码修改前，必须先通过门禁检查。未通过门禁不得修改业务代码。
1. **阶段不可跳过**：必须按 01→07 顺序执行。🟢 简单需求下 02/03 采用**极简模式**（不产出独立文档），但阶段核心逻辑不得完全跳过：02 极简逻辑为需求范围确认 + 4 问判断（是否需 03 方案）；03 极简逻辑为代码探索、关键符号验证、命名冲突预检，并将方案要点合并到 04 任务清单备注。
2. **文档驱动**：每个阶段的产出文档是下一阶段的输入。
3. **04 阶段必须用 Team Agent**（🟢简单需求除外）。
4. **任务清单是唯一真相源**：进度完全由任务清单的勾选状态决定。
5. **任务清单强制审查**：每次任务清单生成/更新后，必须与 03-技术方案 逐项对照。
6. **编译单元注册+构建验证强制**：涉及源文件新增/删除时，必须更新编译单元注册 + 执行构建验证。
7. **构建验证强制**：涉及代码变更的迭代，05 阶段必须先执行构建命令确保 0 编译错误。
8. **逐阶段确认**：每个阶段完成后，必须等待用户明确确认。
9. **能跑则跑，不能则降级**：05 阶段自动生成用例 → 探测环境 → 能自动的执行 → 不能的标记为手动。
10. **命名冲突预检强制**：涉及在已有代码中新增符号时，03 和 04 阶段各执行一次命名冲突扫描。
11. **★ 图优先原则**：涉及架构图、时序图等可视化内容时，优先使用 Mermaid 图表嵌入文档。
12. **★ 独立Agent交叉审查（自审+互审双保险）**：🔴 复杂级 02/03/04 阶段强制启动独立 Agent 交叉审查；🟡 中等级 03/04 强制、02 可选。主 Agent 在需要时先启动独立 Agent，再立即执行自审。详见 `engine/cross-review-protocol.md`。
13. **★ 外部模型审查路由（v2.0.0）**：🔴 复杂级的 02 阶段强制使用，03/04 阶段优先使用。详见 `engine/cross-review-protocol.md` §八。

> 完整流程定义见 `engine/workflow-engine.md`，门禁规则见 `engine/gate-protocol.md`。
> 项目初始化时，引擎会自动生成 `engine-local/`、`project/`、`runtime/` 目录及默认文件。
